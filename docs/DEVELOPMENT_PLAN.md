# 開発計画書 (Development Plan)

## 🎯 目標
今週末でのMVP（Minimum Viable Product）最速リリース。
フロントエンド（React）とバックエンド（FastAPI）を並行開発し、初期段階でAPIスキーマを合意することで手戻りを防ぐ。

## 🔧 運用前提
* Docker Compose はローカル開発と環境差分の吸収に有効だが、MVPの公開要件としては必須ではない。
* 公開時は、詩文さんが保有するドメイン配下で Raspberry Pi サーバーにデプロイする前提とする。
* そのため、開発時は Docker で固めつつ、公開構成は Raspberry Pi 上での実行方式に合わせて最終調整する。

## 📅 タイムライン（週末ハッカソン想定）

### Phase 1: 開発基盤と初期設定の完了（Day 1 午前）
**目的: 落とし穴を潰し、双方がローカルで開発できる状態を作る**
* [x] Docker Composeによるフロント・バック・DBのコンテナ化（開発用。公開時の実行方式は Raspberry Pi 構成に合わせて調整）
* [x] `.env` の分離（`frontend/.env.local`, `backend/.env`）
* [x] DB（PostgreSQL）の永続化設定（Docker Volume）
* [x] AlembicによるDBマイグレーション環境構築（`scores` テーブル作成）
* [x] 静的解析の導入（Frontend: ESLint/Prettier, Backend: Ruff/mypy）
* [x] FastAPI側のCORS設定（`http://localhost:3000` の許可）

### Phase 2: APIスキーマの合意とモック作成（Day 1 午後）
**目的: フロントエンドがUI実装（ハイライト・タイマー）を進められるようにする**
* バックエンド側で、固定のJSONを返すモックAPIを最速でデプロイ（またはローカル提供）する。

**【APIスキーマ設計（合意事項）】**
* `GET /api/v1/stages/generate?level=hard`
```json
{
  "stage_id": "uuid-v4-string",
  "hint": "3桁の数字と4桁の数字をハイフンで繋いだもの",
  "noise_text": "xk3 12-345 999-8888a 123-4567 8b-9999",
  "correct_string": "123-4567"
}

```

* フロントエンドは、ユーザーが入力した正規表現が `noise_text` に対して `correct_string` **のみ**を抽出できた場合に「クリア」と判定する。

### Phase 3: コアロジックの実装（Day 1 夜 〜 Day 2 午前）

**目的: ゲームとして遊べる状態にする**

* **Backend**: 問題自動生成ロジックの実装。正解文字列を生成し、それに「似て非なるダミー」と「完全なノイズ」を混ぜて `noise_text` を構築する。
* **Frontend**: ユーザーの入力に応じたリアルタイムの正規表現ハイライト機能の実装。タイマーの実装。

### Phase 4: アーケードランキング機能の実装（Day 2 午後）

**目的: リプレイ性を高める**

* **Backend**: `POST /api/v1/scores` (スコア登録) と `GET /api/v1/scores/ranking` (トップ10取得) の実装。
* **Frontend**: リザルト画面での3文字ネーム入力と、ランキング表示UIの実装。

### Phase 5: 結合テストとデプロイ（Day 2 夜）

* 通信エラー時のハンドリング、不正な正規表現（構文エラー）入力時のフロントエンドのクラッシュ対策。

#### 📦 Issue 1: 【インフラ/基盤】Docker環境と静的解析ツールの導入
* **詳細**: 
  * `docker-compose.yml` を作成し、React, FastAPI, PostgreSQLが起動するようにする。
  * PostgreSQLのデータは `volumes` で永続化する。
  * フロントエンドにESLint/Prettier、バックエンドにRuff/mypyを導入する。
* **完了条件**: `docker-compose up` で全サービスが立ち上がり、Lintエラーが出ないこと。

#### 📦 Issue 2: 【バックエンド】DBマイグレーションとCORS・環境変数の初期設定
* **詳細**: 
  * `pydantic-settings` を用いて `.env` からDB接続情報を読み込む。
  * FastAPIの `CORSMiddleware` を設定し、フロントからの通信を許可する。
  * Alembicを初期化し、ランキング用の `scores` テーブル（`id`, `player_name`, `clear_time_ms`, `created_at`）を作成するマイグレーションファイルを作成・実行する。
* **完了条件**: DBに `scores` テーブルが存在し、フロントエンドからAPI通信が弾かれないこと。

#### 📦 Issue 3: 【フロントエンド/バックエンド】モックAPIの実装と連携確認
* **詳細**: 
  * 開発計画書のPhase 2で定義したJSONスキーマを返すダミーエンドポイント `GET /api/v1/stages/generate` をFastAPIに実装する。
  * フロントエンドは、そのAPIを叩いてデータを画面に表示するテストを行う。
* **完了条件**: フロントエンドの画面にAPIから取得した `hint` と `noise_text` が表示されること。

#### 📦 Issue 4: 【バックエンド】問題自動生成ロジック（プロシージャル生成）の実装
* **詳細**: 
  * ランダムな正解文字列（例: 電話番号、メアドの一部、日付など）を生成する。
  * 正解を際立たせるための「惜しいダミー文字列」と「ランダムな記号のノイズ」を生成し、一つの長い文字列に結合するロジックを実装する。
* **完了条件**: APIを叩くたびに、異なるランダムな問題データが返ってくること。

#### 📦 Issue 5: 【フロントエンド】プレイ画面（ハイライト・判定ロジック）の実装
* **詳細**: 
  * ユーザーがテキストボックスに入力した正規表現をリアルタイムで評価する。
  * `noise_text` にマッチした箇所をハイライト（背景色変更など）する。
  * マッチした結果が `correct_string` と完全一致した場合、扉が開く演出（クリア画面への遷移）を行う。
  * **注意**: 入力途中で正規表現として不正な構文になった場合の `try-catch` エラーハンドリングを確実に行う（画面を赤くするなどの演出推奨）。
* **完了条件**: 正しい正規表現を入力した時のみクリア判定になること。

#### 📦 Issue 6: 【フルスタック】アーケードランキング機能の実装
* **詳細**: 
  * バックエンドにスコア登録（POST）と取得（GET）のAPIを実装する。
  * フロントエンドに、クリア時の「3文字ネーム入力」と、登録後の「ランキングボード表示」を実装する。
* **完了条件**: 登録したスコアがDBに保存され、ランキング画面で上位順に表示されること。
