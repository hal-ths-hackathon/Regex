Issueタイトル: `[Infra/Backend](13): 本番公開に向けたアーキテクチャ・セキュリティレビュー`

### 背景
アプリケーションをインターネットに公開するにあたり、現状の開発用（ローカル）設定のままデプロイすると予期せぬ脆弱性が露呈する可能性がある。
特にFastAPIのCORS設定、Uvicornの起動オプション、ログ出力の制御など、本番稼働に耐えうる設定へ見直す必要がある。
（※正規表現のReDoS対策については、正誤判定ロジックをフロントエンドに寄せることでバックエンド側の致命的リスクは回避済み）

### 実施内容
- [ ] FastAPIのCORS（`allow_origins`）設定の見直し（`localhost:*` から独自ドメインへの制限）
- [ ] 本番向けの `docker-compose.prod.yml` の作成
  - デバッグ用のVolumeマウント（ホットリロード）の無効化
  - コンテナの再起動ポリシー（`restart: always` または `unless-stopped`）の設定
- [ ] データベース（PostgreSQL）の認証情報と永続化（Volume）のセキュリティ確認
- [ ] `debug.html` などの開発用エンドポイントの本番環境での無効化またはアクセス制限

### 完了条件
- [ ] 開発用と本番用のDocker環境が分離され、コマンド一つで本番用環境が立ち上がること
- [ ] 開発用のデバッグエンドポイントや不要なスタックトレースが本番環境で露出しないこと
- [ ] 本番用の独自ドメインからのリクエストのみAPIが受け付けること（CORS）

### 影響範囲
- `backend/app/main.py`
- `docker-compose.yml` / `docker-compose.prod.yml`
- 環境変数 (`.env`)

### 開発フェーズ
- マイルストーン1「本番公開基盤の構築」
- Phase 2: 本番用設定の適用
